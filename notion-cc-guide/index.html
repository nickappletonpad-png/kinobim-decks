何に使う？（具体例）

例えば、親会社NCNや協力会社の資料を一元管理するデータベース。各箱＝1案件／1資料、プロパティに必要な情報をぶら下げます。

🗄️ 資料データベース（例）
案件AURL PDF IFC 担当・日付
案件BURL PDF 🔗 協力会社
案件CIFC ステータス 🔗 プロジェクト
⏱ 手作業だと大変 → AIと繋げば一瞬 箱を1つずつ作り、URL・PDF・IFCを入れ、ひも付けして、更新し続ける…手作業だと何時間もかかる作業です。 Claude Codeを接続すれば、箱の作成・情報入力・ひも付け・更新をまとめて自動化できます。

※ この「繋ぐ」ための入口が、次からの接続設定です。

3

接続方式は2つ

結論：作り込む／自動化するならトークン方式が本命。

サッと試すだけならコネクタ方式（ログイン承認のみ）。Claude Codeで継続的に読み書き・自動化するならトークン方式（内部インテグレーション）。今吉さんの使い方は後者です。

観点コネクタ方式（OAuth）トークン方式（内部インテグレーション）
セットアップログインして許可するだけインテグレーション作成＋トークン発行
鍵（トークン）の扱い手で扱わない（自動）自分で発行・保管（実質1回表示）
権限の細かさ粗い（クライアント任せ）読取／挿入／更新を個別ON/OFF
アクセス範囲承認ユーザーの閲覧権に連動接続したページ/DBだけに固定
自動化・常駐処理不向き得意（1本のトークンで組込み）
他ツール／スクリプト連携限定的自由（API直結）
複数ユーザーへ配布向く単一ワークスペース向き
取り消しGUIで即接続解除／トークン無効化
向いている人とりあえず試す人CCで作り込む人（＝今吉さん）
4

それぞれの「できる／できない」

サッと試す

コネクタ方式（OAuth）

⭕ できる
  • ログイン承認だけで即接続、鍵管理ゼロ
  • 接続／解除がGUIで完結、最速で試せる
  • Claude.ai／Claude Code に標準搭載
❌ できない・不向き
  • 権限を細かく絞れない
  • 自前スクリプト・常駐処理に組み込みにくい
  • 範囲が承認者の閲覧権に連動（意図せず広く／狭く）
★ 本命

トークン方式（内部インテグレーション）

⭕ できる
  • 読取／挿入／更新を最小権限で設計
  • 接続したページ/DBだけにスコープ厳密固定
  • 1本のトークンで自動化・常駐・他ツール連携
  • ワークスペース単位で管理・監査しやすい
❌ できない・注意
  • トークンは作成時に実質1回しか表示されない（漏洩＝再発行）
  • 使わせたいページを1つずつ「接続」する初期手間
  • トークンは強力な認証鍵。GitHub等に絶対貼らない
  • 多人数配布より単一運用向き
5

トークン方式 セットアップ手順

入口（ここから作成）：

https://www.notion.so/my-integrations

※ Notion設定 →「コネクト」→「インテグレーションを開発・管理」でも同じ画面です。

  1. 「新しいインテグレーション」を作成

    上記URLを開いて新規作成。

  2. 基本情報を設定

    名前（例：KINO-CC）／関連ワークスペースを選択／種類＝内部。

  3. 権限（Capabilities）を設定

    コンテンツを 読む・更新・挿入 をON。ユーザー情報は不要ならOFF。

  4. シークレットをコピー

    送信 → 「内部インテグレーションシークレット」（ntn_… または secret_…）をコピー。

  5. すぐNotionに控える（実質1回表示）

    表示は実質1回きり。Notionの安全なページ／パスワード管理にすぐ保存。紛失＝再発行。

  6. 使わせたいページを「接続」

    対象ページ/DBを開く → 右上 ••• →「接続」→ 作ったインテグレーションを追加。これをしないとAIは中を見られない（＝安全の要）。

  7. CC側にトークンを設定

    Claude Code側にトークンを登録すれば、接続したページを読み書き可能に。

✅ 現状 今回のデモ用は小嶋がフル権限（読取＋挿入＋更新）で発行済み。今吉さんはまず招待メールからログインすればOKです。
6

権限（Capabilities）の中身

権限できること推奨
コンテンツを読むページの閲覧・検索ON
コンテンツを挿入末尾に追記・新規作成ON
コンテンツを更新既存の書き換えON
コメントコメントの読み書き任意
ユーザー情報メンバー情報の取得不要ならOFF
🔒 大事 権限がフルでも、接続したページ/DBしか触れません。共有していないページは見えない＝会社全体が筒抜けにはなりません。
7

実務フロー：議事録 × プロジェクト × Claude Code

★ ここが本番

今吉さんはすでに課金済み。まず作るとよいのが「議事録データベース」です。これをプロジェクトとひも付けると、Claude Codeが効いてきます。

📝
議事録DB
打合せごとに1箱。決定事項・宿題を記録
🔗ひも付け
🗂️
プロジェクトDB
案件ごとに1箱。議事録がぶら下がる
←自動で読む
🤖
Claude Code
案件を指定 → 紐づく議事録を自動把握

この流れで何が起きるか

  1. 議事録DBを作る

    打合せのたびに1箱（タイトル・日付・決定事項・宿題）を追加。

  2. プロジェクトとひも付ける

    各議事録を、該当案件（プロジェクトDBの箱）にリレーションで接続。

  3. Claude Codeに案件を指定

    「この案件の続きやって」と言うだけ。

  4. 議事録を自動で読む

    CCがひも付いた議事録を全部読み、「何が決まって、どこまで進んだか」を把握。

  5. 続きの作業ができる

    要約・次のタスク出し・資料作成・図面準備…毎回説明し直さなくていい。

💡 メリット 議事録とプロジェクトが繋がっていると、AIに毎回背景を説明する手間が消えます。「案件名を言えば、過去の経緯ごと分かっている」状態が作れます。

運用メモ（合意事項）

① トークンは1回表示 → 必ずNotionに控える 機密のため共有ページには載せず、パスワード管理／専用ページで保管。漏洩したら即無効化＆再発行。
② 人を足すときはゲスト 正式メンバーを増やすと席（料金）が増える。お試し段階の追加はゲストで。
③ 勝手にユーザーは増やさない／触れるのは共有ページだけ AIも人も勝手に増やさない。AIが触れるのは明示的に接続したページのみ。
